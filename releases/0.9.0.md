# Public-Access-Agents 0.9.0

## Release classification

- Repository maturity: pre-1.0 baseline
- Compatibility class: initial repository-level compatibility baseline
- Tag: `v0.9.0`
- Release date: 2026-07-13
- License: Apache-2.0

This release establishes the first repository-wide version, changelog, release policy, maturity policy, release tooling, checksums, and tag-driven GitHub Release process.

It does not establish the final `1.0.0` compatibility promise.

## Breaking changes

None.

This is the first repository-level release contract. Existing public paths, rule identifiers, schemas, templates, and executable entry points become the initial pre-1.0 baseline described by [`RELEASE_POLICY.md`](../RELEASE_POLICY.md).

## Normative changes

- Established repository-wide Semantic Versioning rules for normative standards, schemas, templates, tools, stable paths, governance, and release artifacts.
- Defined patch, minor, major, and pre-1.0 classification behavior.
- Defined deprecation windows of at least 90 calendar days for ordinary contracts and 180 calendar days for stable paths and machine-readable contracts.
- Required migration notes for every release.
- Defined the `1.0.0` compatibility gate and release-candidate expectation.
- Added package maturity states and baseline-to-stable promotion requirements.
- Required maturity review records, adoption evidence, compatibility inventory, source review, and independent specialist review for stable promotion.

## Editorial changes

- Added release and maturity navigation to root documentation.
- Added version-specific release and migration guides.
- Clarified that a repository version does not automatically promote every package to stable.

## Tooling changes

- Added release-policy validation to the unified validation pipeline.
- Added deterministic ZIP and TAR.GZ release archive generation.
- Added `SHA256SUMS.txt` generation.
- Added `release-manifest.json` with source commit, artifact sizes, and SHA-256 digests.
- Added tag validation and tag-driven GitHub Release publication.
- Added regression tests for invalid versions, missing release notes, tag mismatches, and incomplete release policy.

## Deprecations

None.

## Migration notes

See [`migrations/0.9.0.md`](migrations/0.9.0.md).

Existing adopters are not required to change their selected standards solely because of this release. They should begin recording the repository version or tag used for future adoption and upgrade decisions.

## Security

- Release publication uses the repository's permanent validation pipeline before artifact creation.
- Release tags must match the canonical `VERSION` file.
- Release archives are built from Git-tracked files with deterministic ordering and normalized metadata.
- SHA-256 checksums and a release manifest are published with each release.
- The release workflow uses scoped `contents: write` permission only for the release job.

## Known limitations

- Most standards packages remain at `baseline`, not `stable`.
- Representative package-level adoption tests are not yet complete.
- The repository currently has one active maintainer.
- Independent specialist coverage remains incomplete for governance, security, schemas, tools, and release automation.
- GitHub ruleset and branch-protection settings require separate administrative verification.
- Release artifacts do not certify adopting projects.

## Validation

Before tagging, run:

```bash
python -m pip install -r tools/validate-schemas/requirements.txt
python tools/validate-all/run_all.py --include-tests
python tools/release/validate_release.py --tag v0.9.0
python tools/release/build_release.py --tag v0.9.0 --output-dir dist --force
```

Verify:

```bash
sha256sum -c dist/SHA256SUMS.txt
```

## Review requirement

This release program changes governance, compatibility, executable tooling, and GitHub Actions behavior. It requires independent specialist review under [`MAINTAINERS.md`](../MAINTAINERS.md).
