# Migration Notes for 0.9.0

## Audience

These notes apply to people who copied standards, templates, schemas, or tools from the repository before the first repository-level version was established.

## Summary

Version `0.9.0` establishes the initial pre-1.0 compatibility baseline.

It does not remove or rename existing public standards packages, schemas, templates, profiles, examples, or executable tool entry points.

## Required actions

Existing adopters should:

1. Record the source repository tag or commit used by their project.
2. Record `0.9.0` in project adoption or evidence documentation when upgrading to this release.
3. Review [`RELEASE_POLICY.md`](../../RELEASE_POLICY.md) for compatibility and deprecation behavior.
4. Review [`MATURITY_POLICY.md`](../../MATURITY_POLICY.md) before treating a package as stable.
5. Continue using major-version schema paths under `schemas/v1/` for durable automation; pin a repository tag or commit when exact schema immutability is required.
6. Run the repository validation tools after copying or updating standards.
7. Review release notes for normative changes before future upgrades.

## Optional improvements

- Use `tools/generate-manifest/generate_manifest.py` to create a version-aware project manifest.
- Use `tools/compose-agents/compose_agents.py` to create a traceable standards bundle.
- Store the repository tag, source commit, and selected package versions with completion evidence.
- Compare future releases against the tag rather than an unversioned branch snapshot.

## Removed contracts

None.

## Deprecated contracts

None.

## Compatibility assumptions

- Existing stable file paths remain present.
- Existing rule identifiers remain present.
- Existing schemas remain available under their rolling and `v1` paths.
- Existing tool entry points remain present.
- Most packages remain `baseline`, not `stable`.

## Validation

From the repository root:

```bash
python -m pip install -r tools/validate-schemas/requirements.txt
python tools/validate-all/run_all.py --include-tests
python tools/release/validate_release.py --tag v0.9.0
```

For an adopting project, re-run its own tests, link checks, schema validation, and project-specific review after updating copied standards.

## Downgrade and rollback

Because `0.9.0` does not remove prior paths, rollback normally means restoring the previously recorded commit or copied standards bundle.

Do not reconstruct the previous state from memory. Use the recorded commit, archive, or source bundle.

## Known limitations

- The repository has not yet published the final `1.0.0` compatibility commitment.
- Package-level adoption testing and stable maturity promotion remain ongoing.
- Repository release checks do not validate an adopting project's environment.
