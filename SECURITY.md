# Security Policy

## Reporting a vulnerability

Do not open a public issue for an exploitable vulnerability involving this repository, its automation, or a published validation tool.

Use GitHub's private vulnerability reporting feature when enabled. If private reporting is unavailable, contact the repository owner through a private GitHub channel before public disclosure.

## Scope

Security reports may include:

- unsafe agent instructions
- examples that expose secrets
- command-injection patterns
- insecure dependency guidance
- bypassable validation rules
- dangerous destructive-action defaults
- supply-chain weaknesses
- repository automation vulnerabilities

## Expectations

A report should include:

- affected file and section
- impact
- reproduction or abuse scenario
- recommended mitigation
- whether public exploitation is known

Do not include real credentials, private keys, tokens, or sensitive third-party data.

## Security claims

These standards are intended to improve secure-development behavior. They do not guarantee that adopting software is secure, compliant, or free of vulnerabilities.
